Desktop vs Servidor

Desktop e servidor compartilham o modelo de Von Neumann[1], mas divergem em quase todos os subsistemas físicos: alimentação, interconexão de CPUs, E/S e gerenciamento. O servidor é projetado para SLA de 99,99% (≈ 4,38 min de downtime por mês).

💡 Analogia: Desktop é um carro esporte (rápido para um motorista). Servidor é um trem de carga: feito para mover toneladas de dados 24/7, com sistemas redundantes e manutenção em movimento.

RAID: Tolerância a Falhas e Desempenho

Originalmente proposto por Patterson, Gibson e Katz em 1988[12], RAID (Redundant Array of Independent Disks) combina N discos físicos numa unidade lógica única para ganhar velocidade, redundância ou ambos.

Hot-swap & Rebuild 🕵️

Em servidores, os discos são hot-swap: se um falhar em RAID 1, 5 ou 10, o técnico troca-o com o servidor ligado e a controladora inicia o rebuild. Durante o rebuild há janela de risco — perder um segundo disco no RAID 5 nesse momento = perda total.

Como o RAID 5 reconstrói dados

A paridade é um XOR bit-a-bit entre os blocos de dados. Se A, B, C são blocos e P é a paridade:

P = A ⊕ B ⊕ C Se C for perdido: C = A ⊕ B ⊕ P (recalculado em tempo real durante leitura)

É por isso que o RAID 5 tem boa leitura mas escrita pesada: cada gravação exige ler o bloco antigo, ler a paridade antiga, calcular a nova paridade e gravar os dois — o famoso RAID 5 write penalty.

Por que IOPS de escrita despenca em RAID 5/6?

Cada escrita lógica em RAID 5 gera 4 operações físicas (read-modify-write): lê dado antigo, lê paridade antiga, escreve dado novo, escreve paridade nova. Em RAID 6 são 6 (P + Q). Por isso bancos de dados OLTP de alta concorrência preferem RAID 10, que tem penalidade de apenas 2×.

A Lei de Amdahl[10] impõe um limite teórico ao ganho de paralelizar discos. Se uma fração P do tempo é paralelizável e N é o nº de discos:

Speedup(N) = 1 / [ (1 − P) + P/N ] Quando N → ∞, Speedup → 1/(1−P). A parte serial é o teto absoluto.

📄 Exemplo: servidor WWW da Aula 9

O Prof. Koide cita em aula[17] um servidor servindo páginas de 8 KB, com disco 10.000 rpm (seek 8 ms) e taxa de transferência de 10 MB/s. Cerca de 99,3% do tempo de resposta é latência rotacional + seek (i.e., serial por disco) e apenas 0,7% é transferência paralelizável.

Se P = 0,007 (só 0,7% é paralelizável entre discos), mesmo um RAID com infinitos discos teria speedup máximo:

Speedup máx. = 1 / (1 − 0,007) ≈ 1,007×

Conclusão: para essa carga, paralelizar discos não ajuda. Trocar HDD por SSD (que ataca a parte serial) ou aumentar cache de RAM resolve. Esse é o ensinamento da lei: otimize a fração dominante, não a paralelizável.

Memória ECC: Detecção e Correção de Erros

Partículas subatômicas (raios cósmicos secundários, contaminação alfa de solda) podem inverter um bit de DRAM sem aviso — o famoso soft error. Em servidor 24/7, isso é estatisticamente certo[11]. ECC detecta e corrige em tempo real.

🔍 Como o circuito sabe qual bit virou?

Na leitura, recalculam-se as 4 paridades. Cada paridade discordante "vota" em um bit do síndrome (s3 s2 s1 s0). O valor decimal do síndrome é exatamente a posição do bit corrompido. Síndrome = 0 ⇒ sem erro.

Ex.: síndromes = (0,1,0,1) ⇒ posição binária 0101 = 5 ⇒ D2 está corrompido O hardware inverte o bit da posição 5 e entrega o dado correto à CPU.

Com o bit extra global, detecta-se ainda 2 erros simultâneos (DED) — embora não consiga corrigir, ao menos a falha é sinalizada (MCE — Machine Check Exception) em vez de causar corrupção silenciosa.

O estudo seminal do Google em sua frota[11] analisou bilhões de DIMMs durante 2,5 anos. Resultados que mudaram a forma de pensar memória de servidor:

Onde os erros vêm?

  • Raios cósmicos secundários: nêutrons atravessam a atmosfera e ionizam o silício. Mais frequentes em altitude (servidores em Denver têm taxa > nível do mar).
  • Contaminação alfa: traços de urânio/tório na solda emitem partículas α que invertem bits adjacentes.
  • Rowhammer: ataque deliberado — acessos repetidos a uma linha de DRAM induzem bit-flips em linhas vizinhas (CVE-2015-0565).
  • Desgaste: envelhecimento de células, especialmente em DDR4/DDR5 com densidades crescentes.

O caso "supereleitor" da Bélgica

Eleições municipais de Schaerbeek, 2003: o candidato Maria Vindevoghel recebeu 4.096 votos extras. Investigação concluiu: um único bit flipou no bit de posição 13 (2¹² = 4096) do contador, provavelmente por raio cósmico. O sistema não tinha ECC. É um dos casos publicamente confirmados de soft error com impacto real.

Atenção: DDR5 desktop tem "on-die ECC", mas não é ECC verdadeiro. Ele corrige falhas internas da célula antes de expor o dado, mas não protege contra erros no caminho DIMM → CPU. Só o DIMM ECC registrado (RDIMM) de servidor cobre todo o datapath.

📍 Onde a ECC fica fisicamente

O bit extra de paridade ocupa um chip a mais no PCB do DIMM (8 chips de dado + 1 ECC = 9 visíveis). O cálculo Hamming é feito pelo controlador de memória integrado (IMC) dentro da CPU — não pelo DIMM. Para o SO, o erro corrigido aparece como evento no /var/log/mcelog (Linux) ou WHEA log (Windows). É possível monitorar a tendência: muitos CEs num DIMM = sinal para troca preventiva antes que vire UE.

    🔄 Coerência de cache: o protocolo MESI

    Quando 8 cores compartilham a L3 e 2 sockets compartilham a RAM, é preciso garantir que todos vejam a mesma versão de cada linha. O protocolo MESI define 4 estados por linha de cache:

    EstadoSignificadoPode ler?Pode escrever?
    M — ModifiedLinha alterada, só este core temSimSim, sem invalidar
    E — ExclusiveLimpa, só este core temSimSim, vira M
    S — SharedLimpa, vários cores têmSimNão — precisa invalidar os outros
    I — InvalidLinha não vale (foi invalidada)Não — gera missNão

    Intel acrescenta o estado F (Forward) em MESIF: numa linha Shared, um único core fica responsável por servir cópias aos outros, reduzindo tráfego no barramento. AMD usa MOESI, adicionando Owned para suportar escritas sem write-back imediato à RAM.

    O SO entrega à aplicação um espaço de endereços virtual. A MMU traduz cada acesso para o endereço físico via tabelas de página — operação cara, então cacheada na TLB (Translation Lookaside Buffer)[4].

    Huge page2 MB1 entrada TLB cobre 512× mais memóriaFragmentação interna

    📐 Custo de um miss na TLB

    x86-64 usa paginação em 4 níveis (PML4 → PDPT → PD → PT). Cada miss na TLB exige até 4 acessos à RAM para fazer a tradução — o famoso page walk. Por isso aplicações HPC e bancos de dados configuram transparent huge pages (THP) ou libhugetlbfs.

    t_acesso = t_TLB + (taxa_miss × t_pagewalk) Com THP, taxa_miss cai > 90% em workloads de grande footprint.

    🔗 TLB + Cache + NUMA juntos

    O hardware acessa TLB e L1 em paralelo (VIPT — Virtually Indexed, Physically Tagged) para esconder a latência da tradução. Em servidor NUMA, a TLB também rastreia qual nó hospeda cada página — alocação errada (página em RAM remota) custa > 60 ns por acesso e quase nenhum scheduler genérico corrige isso sem hinting do aplicativo (numactl --membind, libnuma).

    🛡️ Anéis de Proteção e o "Ring -1"

    x86 define 4 anéis de privilégio. SOs usam o Ring 0 (kernel) e Ring 3 (usuário). Ring 1 e 2 ficaram historicamente sem uso. O problema: como o hypervisor roda acima do SO da VM, precisaria de um anel "mais interno" que o Ring 0 — daí Intel VT-x e AMD-V introduziram o Ring −1 (também chamado VMX root mode)[5].

    Ring 3 — Aplicações
    Ring 0 — Kernel
    −1

    Sem VT-x/AMD-V, técnicas de binary translation (VMware até 2006) ou paravirtualização (Xen) eram necessárias. Hoje qualquer CPU de servidor traz extensões em hardware: VMCS (Intel) ou VMCB (AMD), tradução de endereços por EPT/NPT, e instruções como VMENTER/VMEXIT para chavear contexto.

    WSL 2 — virtualização "leve"

    O Windows Subsystem for Linux 2 não é tradução de syscalls (como WSL 1 era); é um kernel Linux real rodando em uma VM utility leve gerenciada pelo Hyper-V[17]. Boot quase instantâneo, integração com o sistema de arquivos do Windows via 9P, e desempenho próximo do nativo — virtualização Tipo 1 invisível ao usuário.

    Contêineres usam recursos do próprio kernel Linux (namespaces isolam processo/rede/PID; cgroups limitam CPU/RAM/IO). Não há kernel virtualizado, então um contêiner é tão leve quanto um processo[6].

    • Banco de dados: PostgreSQL, MySQL, MongoDB, Redis.
    • Mensageria/cache: Kafka, RabbitMQ, Memcached.

    Para cada requisição, esse processo aciona todas as camadas abaixo: lê dados do disco (⑦), aloca buffers em RAM (⑥), executa em vCPUs (⑤), passa pelo hypervisor (④), entrega via rede (⑧). Otimizar performance significa entender essa cadeia inteira.

    O efeito cascata de uma só requisição HTTP

    Uma única requisição "GET /produto/123" num e-commerce moderno típico atravessa: load balancer (⑧) → ingress controller no pod K8s (②) → container app (②) → query SQL no DB pod (②) → driver NUMA-aware (⑥) → leitura de página em SSD NVMe via RAID controller (⑦) → ECC valida o byte que voltou (⑥) → resposta JSON volta toda a pilha em ~10 ms. Cada camada custou microsegundos.
